# 电子信息制造业数智化平台详细设计 v3 - .NET 专项方案

> 版本：v3.0 .NET 详细设计  
> 日期：2026-07-06  
> 依据文档：`架构/系统架构-v3-.NET专项方案.md`  
> 技术路线：`.NET + React + SQL Server`  
> 第一版目标：完成需求、调研、数据目录、权限、规则、工具、智能体、协同入口、审计闭环  
> 后续扩展：Python AI Runtime、RAG、模型评测、算法服务、本地大模型

## 1. 设计目标

本详细设计用于指导第一版系统开发落地。第一版不追求完整低代码平台、复杂多智能体自治或完整知识资产运营，重点交付一个可上线、可管控、可审计的数智化智能体工厂。

第一版必须满足：

| 目标 | 说明 |
| --- | --- |
| 用户身份准确 | Web、飞书、钉钉入口均能识别到平台用户 |
| 用户权限准确 | 数据源、表、字段、规则、工具、智能体全链路校验 |
| 用户输入输出准确 | 输入有上下文约束，输出有脱敏、依据和审计 |
| 开发者模块完整 | 数据源、目录、规则、工具、智能体、Workspace、发布流程闭环 |
| 管理员治理完整 | 租户、用户、角色、菜单、权限、发布审核、审计日志 |
| 协同平台闭环 | 飞书或钉钉至少一个平台完成消息接入、任务执行、结果返回 |
| 审计可追踪 | 任意一次智能体任务可通过 trace_id 追溯完整链路 |

第一版主链路：

```text
管理员配置租户、用户、角色、菜单、数据权限
  -> 企业用户提交场景需求
  -> FDE/精益人员记录调研和数据摸底
  -> 开发者配置数据源、数据目录、规则、工具、智能体
  -> Workspace 沙箱测试
  -> 管理员审核发布
  -> 用户通过 Web/飞书/钉钉输入问题
  -> 系统计算权限并构建安全上下文
  -> 智能体调用授权规则和工具
  -> 系统脱敏、校验、返回结果
  -> 审计全链路记录
```

## 2. 技术实现总览

### 2.1 技术栈

| 层级 | 技术 |
| --- | --- |
| 前端 | React + TypeScript + Vite |
| UI | Ant Design 或 Semi Design |
| 后端 | .NET LTS + ASP.NET Core Web API |
| ORM | EF Core |
| 复杂查询 | Dapper |
| 主数据库 | SQL Server |
| 缓存 | Redis |
| 对象存储 | MinIO / OSS |
| 任务调度 | Hangfire，复杂定时任务可引入 Quartz.NET |
| 日志 | Serilog |
| 链路追踪 | OpenTelemetry |
| API 文档 | Swagger / OpenAPI |
| 参数校验 | FluentValidation |
| 对象映射 | Mapster，复杂场景可手写映射 |
| LLM 接入 | .NET LLMAdapter |
| 后续 AI 扩展 | Python FastAPI AI Runtime |

### 2.2 应用划分

第一版建议采用一个 React 前端工程和一个 .NET 后端工程，通过菜单和路由区分不同端。

```text
前端：
  admin      管理后台
  enterprise 企业端
  developer  开发者端
  fde        FDE 工作台

后端：
  Platform API
  Webhook API
  Worker
```

前端第一版可以单体工程：

```text
src/
  apps/
    admin/
    enterprise/
    developer/
    fde/
  components/
  routes/
  services/
  stores/
  permissions/
```

后端第一版采用模块化单体：

```text
src/
  Bidong.AgentFactory.Api
  Bidong.AgentFactory.Application
  Bidong.AgentFactory.Domain
  Bidong.AgentFactory.Infrastructure
  Bidong.AgentFactory.Worker
  Bidong.AgentFactory.Contracts
```

### 2.3 后端分层职责

| 层 | 职责 | 禁止事项 |
| --- | --- | --- |
| Api | Controller、鉴权、路由、Webhook、统一异常 | 不写复杂业务逻辑 |
| Application | 用例编排、事务、权限、发布审核、任务创建 | 不直接依赖具体 SDK |
| Domain | 领域对象、状态机、领域规则 | 不依赖 EF Core、Redis、LLM SDK |
| Infrastructure | SQL Server、Redis、MinIO、LLM、飞书、钉钉、外部数据源 | 不绕过 Application 调业务 |
| Worker | Hangfire Job、后台任务、异步执行 | 不直接暴露外部 API |
| Contracts | DTO、请求响应、事件契约 | 不放业务实现 |

### 2.4 基础约定

| 项 | 约定 |
| --- | --- |
| API 前缀 | `/api/v1` |
| Webhook 前缀 | `/webhooks` |
| 主键 | `uniqueidentifier`，应用层生成 Guid |
| 租户隔离 | 核心业务表必须包含 `tenant_id` |
| 审计字段 | `created_by`、`created_at`、`updated_by`、`updated_at` |
| 软删除 | `is_deleted`、`deleted_at`、`deleted_by` |
| 并发控制 | 关键配置表使用 `row_version` |
| 时间 | 后端统一 UTC 存储，前端按时区展示 |
| API 返回 | 统一响应结构 |
| trace | 所有请求、任务、工具、模型调用传递 `trace_id` |

统一 API 返回：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {},
  "traceId": "xxx"
}
```

错误返回：

```json
{
  "success": false,
  "code": "PERMISSION_DENIED",
  "message": "无权限访问该数据",
  "data": null,
  "traceId": "xxx"
}
```

## 3. 数据库通用设计

### 3.1 通用字段

核心业务表统一包含：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | uniqueidentifier | 主键 |
| tenant_id | uniqueidentifier | 租户 ID |
| created_by | uniqueidentifier | 创建人 |
| created_at | datetime2 | 创建时间 |
| updated_by | uniqueidentifier | 更新人 |
| updated_at | datetime2 | 更新时间 |
| is_deleted | bit | 是否删除 |
| deleted_by | uniqueidentifier null | 删除人 |
| deleted_at | datetime2 null | 删除时间 |
| row_version | rowversion | 并发控制 |

系统级表可不带 `tenant_id`，例如系统菜单、系统权限字典、全局配置。

### 3.2 命名约定

| 类型 | 约定 | 示例 |
| --- | --- | --- |
| 表名 | 模块前缀 + 业务名 | `identity_users` |
| 主键 | `id` | `id` |
| 外键 | `xxx_id` | `tenant_id` |
| 状态 | `status` | `enabled`、`disabled` |
| JSON 配置 | `config_json` | 智能体模型配置 |
| 摘要字段 | `summary` | 结果摘要、SQL 摘要 |

### 3.3 事务边界

| 场景 | 事务要求 |
| --- | --- |
| 用户角色变更 | 用户角色、权限缓存失效、审计必须一致 |
| 数据权限变更 | 权限记录、缓存失效、审计必须一致 |
| 规则发布 | 新版本、生效版本、发布记录必须一致 |
| 智能体发布 | 智能体版本、绑定关系、发布记录必须一致 |
| 任务执行 | 任务状态和审计允许最终一致，但状态必须可恢复 |

## 5. 数据权限模块

### 5.1 权限模型

数据权限分为四层：

```text
数据源权限
  -> 表权限
  -> 字段权限
  -> 行级条件预留
```

第一版必须实现数据源、表、字段权限。行级权限可预留，但不强制上线。

### 5.2 关键表

`auth_data_permissions`

| 字段 | 说明 |
| --- | --- |
| id | 权限 ID |
| tenant_id | 租户 ID |
| subject_type | user / role |
| subject_id | 用户 ID 或角色 ID |
| resource_type | data_source / table / column |
| resource_id | 资源 ID |
| action | read / execute |
| effect | allow / deny |
| condition_json | 行级条件预留 |
| status | enabled / disabled |

### 5.3 有效权限计算

```text
用户直接权限
  + 用户角色权限
  + 智能体绑定范围
  - deny 权限
  = EffectivePermission
```

第一版规则：

- deny 优先于 allow。
- 用户权限和智能体绑定范围取交集。
- Prompt 只能注入交集范围内的数据目录。
- 工具执行前必须再次计算或读取权限快照。

### 5.4 权限快照

每次智能体任务创建时生成权限快照：

`task_permission_snapshots`

| 字段 | 说明 |
| --- | --- |
| task_id | 任务 ID |
| user_id | 用户 ID |
| tenant_id | 租户 ID |
| agent_id | 智能体 ID |
| data_scope_json | 本次任务有效数据范围 |
| tool_scope_json | 本次任务有效工具范围 |
| rule_scope_json | 本次任务有效规则范围 |
| created_at | 创建时间 |

用途：

- 保证任务执行过程可追溯。
- 审计时可以还原当时权限。
- 避免任务执行中途权限变化导致链路不可解释。

### 13.3 执行流程

```text
1. 校验用户和租户状态
2. 校验用户是否可访问智能体
3. 加载智能体当前发布版本
4. 计算 EffectivePermission
5. 保存权限快照
6. 加载授权后的数据目录、规则、工具
7. 构建 Prompt
8. 调用 LLM
9. 判断是否需要工具调用
10. 工具执行前二次权限校验
11. 执行规则或工具
12. 将工具结果返回模型或直接汇总
13. AnswerGuard 输出校验
14. 结果脱敏
15. 保存结果和审计
```

### 13.4 Prompt 结构

```text
系统角色：
  你是企业数智化智能体，只能基于授权数据、规则和工具回答。

权限约束：
  你不能请求或输出未授权数据。
  如果用户询问无权限数据，必须说明无权限。

可用数据目录：
  仅注入用户有权限的表和字段。

可用规则：
  仅注入用户有权限的规则。

可用工具：
  仅注入用户有权限的工具名称、用途、参数。

输出要求：
  给出结论、依据、口径、trace_id。
```

### 13.5 AnswerGuard

输出前检查：

| 检查 | 说明 |
| --- | --- |
| 敏感字段 | 手机号、身份证、员工信息等脱敏 |
| 无权限字段 | 不允许返回 |
| 工具结果范围 | 不超过授权范围 |
| 幻觉提示 | 无依据时提示无法确认 |
| 格式检查 | 按输出 Schema 返回 |

后端必须再次校验，前端隐藏不能作为安全依据。

### 19.3 关键页面

#### 开发者 Workspace

```text
左侧：智能体资产树

底部：发布检查结果
```

必须支持：

- 选择模拟用户。
- 查看有效数据范围。
- 查看可用规则和工具。
- 执行沙箱测试。
- 查看 trace。

#### 数据权限配置页

页面区域：

```text
左侧：用户/角色
中间：数据源/表/字段树
右侧：权限预览
```

必须支持：

- 按用户配置。
- 按角色配置。
- 字段级勾选。
- 模拟用户预览。


## 23. 关键风险与处理

| 风险 | 影响 | 处理 |
| --- | --- | --- |
| 权限模型过粗 | 智能体越权访问数据 | 第一版必须字段级权限 |
| LLM 幻觉 | 输出不准确 | 限制工具和规则，输出依据 |
| SQL 工具失控 | 慢查询或越权 | SQLGuard、只读账号、限行、超时 |
| 协同用户识别错误 | 用户权限错配 | 使用 open_id/union_id/user_id 映射 |
| 数据目录注释不足 | 智能体理解错误 | 发布检查强提示 |
| 规则口径不一致 | 输出结果错误 | 规则版本、测试、发布审核 |
| 审计不完整 | 责任不可追溯 | trace_id 强制贯穿 |
| 第一版范围过大 | 交付延期 | 低代码、RAG、Python 算法后置 |

## 24. 第一版不做内容

| 内容 | 原因 | 后续方向 |
| --- | --- | --- |
| 完整低代码平台 | 开发成本高，偏离核心闭环 | 第三期平台化 |
| 多智能体自治 | 权限和可控性风险高 | 第二期评估 |
| Python Agent Runtime | 第一版可由 .NET 内置轻量实现 | 第二期拆分 |
| RAG 知识库 | 第一版重点是数据权限和工具闭环 | 第二期建设 |
| 本地大模型 | 硬件和模型效果不确定 | 客户强需求时专项 |
| 复杂 VSM/鱼骨图 | 前端绘图成本高 | FDE 工作台增强 |
| 自动规则优化 | 容易影响业务口径 | 先做建议，不自动发布 |

## 25. 结论

本详细设计建议第一版采用 `.NET + React + SQL Server` 完成主业务系统闭环。

第一版核心不是模型能力复杂度，而是：

```text
身份准确
权限准确
数据目录准确
规则口径准确
工具执行可控
智能体输出可控
协同入口可用
审计链路完整
```

后续 Python AI Runtime、RAG、模型评测、制造业算法、本地大模型应作为第二期或专项能力引入，避免第一版技术范围失控。
