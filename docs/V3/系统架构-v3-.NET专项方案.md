# 电子信息制造业数智化平台系统架构 v3 - .NET 专项方案

> 版本：v3.0 .NET 专项方案  
> 日期：2026-07-06  
> 技术路线：.NET + React + SQL Server  
> 适用范围：第一版主业务系统、权限治理、智能体工厂、协同平台接入、审计闭环  
> 扩展方向：后续通过 Python AI Runtime 扩展 RAG、算法、模型评测、本地大模型能力

## 1. 方案结论

`.NET + React + SQL Server` 适合本项目第一版主系统建设，尤其适合企业私有化部署、微软技术栈团队、SQL Server 既有环境、AD/LDAP 集成和企业级权限审计场景。

本方案建议：

```text
前端：React + TypeScript + Vite + Ant Design
后端主系统：.NET LTS + ASP.NET Core Web API
数据库：SQL Server
缓存：Redis
对象存储：MinIO / OSS
任务调度：Hangfire 或 Quartz.NET
协同平台：飞书或钉钉 Webhook + OpenAPI
智能体运行：第一版内置 .NET Agent Runtime
AI 扩展：第二期拆分 Python AI Runtime
部署：Docker Compose 起步，后续 Kubernetes 或企业 Windows/Linux 混合部署
```

核心判断：

| 判断项 | 结论 |
| --- | --- |
| 是否适合第一版 | 适合 |
| 是否适合权限、租户、审计 | 适合 |
| 是否适合企业后台和私有化交付 | 适合 |
| 是否适合作为唯一 AI 技术栈 | 不建议 |
| 是否需要 Python | 第一版可不强依赖，第二期建议引入 |
| 是否需要本地大模型 | 第一版不强制，按客户保密要求预留 |

## 2. 适用场景

### 2.1 推荐采用 .NET 路线的情况

| 场景 | 说明 |
| --- | --- |
| 团队 C#/.NET 能力强 | 可降低开发、排障和维护成本 |
| 客户已有 SQL Server | 数据库运维、备份、权限体系更容易落地 |
| 客户有 Windows Server / AD | 可对接企业账号和内网基础设施 |
| 项目以私有化交付为主 | .NET + SQL Server 在传统企业接受度高 |
| 权限和审计要求高 | ASP.NET Core 中间件、认证授权、审计体系成熟 |
| 后台管理功能重 | 用户、租户、角色、菜单、审批、审计适合 .NET 实现 |

### 2.2 不建议只用 .NET 覆盖的部分

| 部分 | 原因 | 建议 |
| --- | --- | --- |
| RAG 知识库深度能力 | Python 文档解析、向量化、Rerank 生态更成熟 | 第二期拆 Python RAG 服务 |
| 制造业算法模型 | Python 机器学习、优化算法、视觉算法生态更强 | Python 算法服务 |
| Prompt 批量评测 | Python 更适合快速接模型和评测框架 | Python 评测服务 |
| 本地大模型推理 | vLLM、Ollama、TensorRT-LLM 主要由 Python 生态承接 | Python Model Gateway |
| 复杂脚本工具 | 统计分析、报表生成、数据摸底 Python 更高效 | Python 脚本沙箱 |

### 3.2 第一版架构形态

第一版采用模块化单体，不建议拆微服务。

原因：

| 原因 | 说明 |
| --- | --- |
| 交付效率高 | 第一版重点是业务闭环，不是分布式治理 |
| 权限一致性好 | 用户、租户、数据权限、审计在一个进程内更容易保证 |
| 部署复杂度低 | 私有化交付时更容易安装、升级、排障 |
| 事务处理简单 | 规则发布、权限变更、审计写入可保持一致 |
| 后续可拆分 | 通过清晰模块边界为服务化演进预留空间 |

## 7. 权限架构

权限是本项目的核心风险点，必须在 .NET 后端强制实现。

### 7.1 权限层级

```text
租户权限
  -> 用户所属租户
  -> 企业/基地/服务商隔离

功能权限
  -> 菜单
  -> 按钮
  -> API

资源权限
  -> 智能体
  -> 数据源
  -> 表
  -> 字段
  -> 规则
  -> 工具

运行时权限
  -> Prompt 上下文过滤
  -> 工具执行前校验
  -> SQL 安全校验
  -> 返回结果脱敏
```

### 7.2 .NET 权限实现建议

| 能力 | .NET 实现方式 |
| --- | --- |
| 登录态 | JWT / Cookie / OIDC |
| API 鉴权 | ASP.NET Core Authentication |
| API 授权 | ASP.NET Core Authorization |
| 自定义权限 | PermissionHandler |
| 租户上下文 | TenantMiddleware |
| 数据权限 | DataPermissionService |
| 工具权限 | ToolPermissionGuard |
| 输出脱敏 | AnswerGuard |
| 审计 | AuditMiddleware + AuditService |

### 7.3 智能体权限链路

```text
用户输入
  -> 获取 UserContext
  -> 获取 TenantContext
  -> 校验用户是否可访问智能体
  -> 计算用户数据权限
  -> 计算智能体绑定数据范围
  -> 求交集得到 EffectivePermission
  -> ContextBuilder 只注入授权数据目录
  -> ToolExecutor 执行前再次校验
  -> SQLGuard 检查 SQL 安全
  -> AnswerGuard 检查输出
  -> AuditService 写入审计
```

关键要求：

| 要求 | 说明 |
| --- | --- |
| 权限不能只做前端隐藏 | 后端必须拦截 |
| Prompt 不能包含无权限表字段 | 防止模型知道不该知道的数据结构 |
| SQL 执行前必须二次校验 | 防止模型生成越权 SQL |
| 返回结果必须脱敏 | 防止敏感字段泄露 |
| 权限变更要尽快生效 | Redis 缓存需支持失效机制 |

## 8. 智能体运行方案

### 8.1 第一版 .NET 内置 Agent Runtime

第一版建议在 .NET 主系统中实现轻量 Agent Runtime，降低系统复杂度。

```text
AgentRuntime
  ContextBuilder
  PromptBuilder
  PermissionGuard
  LlmAdapter
  ToolPlanner
  ToolExecutor
  RuleExecutor
  AnswerGuard
  TraceRecorder
```

### 8.2 执行流程

```text
用户问题
  -> 创建 Task
  -> 加载 Agent 配置
  -> 计算 EffectivePermission
  -> 加载授权后的数据目录、规则、工具
  -> 构建 Prompt
  -> 调用 LLM
  -> 解析模型工具调用意图
  -> 执行规则或工具
  -> 汇总结果
  -> 脱敏和安全检查
  -> 返回用户
  -> 写入 task_result 和 audit
```

### 8.3 LLMAdapter

LLM 供应商必须通过适配器隔离。

```csharp
public interface ILlmAdapter
{
    Task<LlmResponse> ChatAsync(LlmRequest request, CancellationToken cancellationToken);
}
```

适配目标：

```text
OpenAI-compatible API
Azure OpenAI
通义千问
智谱
DeepSeek
企业私有模型 API
后续 Python Model Gateway
```

### 8.4 工具运行

第一版工具类型：

| 工具类型 | 说明 |
| --- | --- |
| SQL 工具 | 查询授权数据库，默认只读 |
| HTTP 工具 | 调用企业系统 API |
| Rule 工具 | 调用规则引擎计算业务指标 |

SQL 工具要求：


### 11.3 .NET 与 Python 调用关系

```text
.NET TaskService
  -> Python AI Runtime API
  -> Python 返回结构化结果
  -> .NET AnswerGuard
  -> .NET AuditService
  -> 返回用户
```

原则：

| 原则 | 说明 |
| --- | --- |
| .NET 负责权限 | Python 不能自行扩大权限 |
| .NET 负责审计 | Python 只返回执行过程摘要 |
| .NET 负责发布 | Python 只能生成建议 |
| Python 负责 AI 能力 | 模型、RAG、算法、评测由 Python 承接 |

## 15. 风险与规避

| 风险 | 说明 | 规避 |
| --- | --- | --- |
| 把 .NET 当成唯一 AI 技术栈 | 后续 RAG、评测、算法扩展成本高 | 第一版内置，第二期拆 Python |
| SQL Server 授权成本 | 私有化和多租户规模化时成本增加 | 前期确认客户授权策略 |
| 权限只做菜单级 | 智能体可能越权访问数据 | 强制数据源、表、字段、工具、输出全链路校验 |
| LLM 直接访问数据库结构 | 可能泄露无权限表字段 | ContextBuilder 按权限过滤 |
| SQL 工具失控 | 可能慢查询、越权或破坏数据 | 只读账号、SQLGuard、限行、超时 |
| Webhook 超时 | 飞书/钉钉消息失败 | 快速 ACK + Hangfire 异步执行 |
| 审计不完整 | 出问题无法追责 | trace_id 贯穿 API、任务、工具、模型 |
| 过早微服务化 | 增加联调和部署复杂度 | 第一版模块化单体 |

## 16. 最终建议

如果当前团队具备 .NET 能力，且项目更偏企业私有化部署，则推荐第一版采用：

```text
.NET LTS + ASP.NET Core
React + TypeScript
SQL Server
Redis
MinIO
Hangfire
OpenTelemetry
```

第一版重点不是追求复杂 AI 框架，而是完成：

```text
用户身份准确
数据权限准确
开发者配置完整
智能体执行可控
协同平台闭环
审计链路完整
```

后续 AI 能力通过 Python 扩展，而不是让 .NET 主系统承担所有模型、RAG、算法和评测能力。这样既能保证第一版可交付，也能避免后续 AI 能力扩展被技术栈锁死。
