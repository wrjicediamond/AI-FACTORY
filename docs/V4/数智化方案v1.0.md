# 电子信息制造业企业智能化项目系统整体流程与模块方案 v2

> 定位：面向立项、开发和交付的可实现性方案  
> 依据：`系统整体流程与模块方案-v1.md`、`.NET 专项架构方案`、`.NET 详细设计方案`  
> 技术路线：React + TypeScript + ASP.NET Core Web API + SQL Server + Redis + MinIO/OSS + Hangfire  
> 设计重点：先讲清楚系统为什么做、做什么，再讲用户端与开发端如何隔离、如何协作、每个模块如何落地实现。

## 1. 项目背景

电子信息制造业企业在推进智能化和数智化过程中，通常已经积累了 MES、ERP、SCADA、质量系统、设备系统、Excel 台账等大量数据，但这些数据分散在不同系统里，字段含义不统一，指标口径不一致，权限边界也不清晰。企业业务人员提出的问题往往是模糊的，例如“想提升良率”“想看设备异常原因”“想知道某条产线成本为什么高”，这些问题不能直接交给大模型处理。

如果让大模型直接访问数据库，会带来三个风险：

| 风险 | 说明 |
| --- | --- |
| 数据越权 | 用户可能看到不该看的产线、成本、客户、订单或财务数据 |
| 口径错误 | AI 可能按自己的理解计算良率、OEE、成本、利润等指标 |
| 结果不可追溯 | 无法说明答案用了哪些数据、规则、工具和模型 |

因此，本项目要建设的不是一个简单的 AI 问答入口，而是一个面向电子信息制造业企业的“可配置、可授权、可测试、可审计”的智能化平台。平台把企业业务需求转化为可运行的智能体能力，让 AI 只能在授权数据、授权规则和授权工具范围内工作。

## 2. 系统核心功能总结

系统核心功能可以概括为“一个闭环、两个端、三个中心、四类资产”。

### 2.1 一个闭环

```text
用户提出需求
  -> 系统结构化需求
  -> 开发端评估数据和规则
  -> 开发端配置数据目录、权限、规则、工具、智能体
  -> Workspace 沙箱测试
  -> 发布审核
  -> 用户使用智能体
  -> 系统返回有依据、有权限说明、有 trace_id 的结果
  -> 用户反馈
  -> 平台沉淀模板、规则、工具和交付报告
```

这个闭环保证每个智能化场景都能从“模糊想法”走到“可交付能力”，并且过程可追踪。

### 2.2 两个端

| 端 | 面向对象 | 核心目标 |
| --- | --- | --- |
| 用户端 | 企业业务人员、管理人员、产线负责人 | 低门槛提需求、看进度、用智能体、看结果、做反馈 |
| 开发端 | 平台开发者、实施人员、FDE、数据工程人员 | 把需求变成数据、规则、工具和智能体配置，并完成测试发布 |

### 2.3 三个中心

| 中心 | 作用 |
| --- | --- |
| 流程中心 | 管理需求状态、项目状态、审核、发布、反馈和验收 |
| 权限中心 | 管理租户、用户、角色、菜单、API、数据、字段、工具、智能体权限 |
| 智能体运行中心 | 执行智能体任务，完成权限计算、上下文构建、模型调用、工具调用、输出检查和审计 |

### 2.4 四类资产

| 资产 | 说明 |
| --- | --- |
| 需求资产 | 需求单、需求.md、需求版本、补充材料、验收标准 |
| 数据资产 | 数据源、数据目录、字段注释、敏感级别、数据权限 |
| 能力资产 | 业务规则、SQL/HTTP/Rule 工具、智能体配置、测试用例 |
| 交付资产 | 测试报告、发布记录、运行审计、交付报告、可复用模板 |

## 3. 总体实现架构

第一版建议采用一个 React 前端工程和一个 ASP.NET Core 后端工程，通过路由、菜单和权限划分用户端、开发端和管理端。后端采用模块化单体，便于第一版快速交付，同时保证权限、事务和审计链路一致。

```text
React 前端
  -> 用户端路由 /enterprise
  -> 开发端路由 /developer
  -> 管理端路由 /admin

ASP.NET Core Web API
  -> Identity 模块
  -> Authorization 模块
  -> Workflow 模块
  -> Demand 模块
  -> DataCatalog 模块
  -> RuleEngine 模块
  -> ToolRuntime 模块
  -> AgentStudio 模块
  -> Workspace 模块
  -> AgentRuntime 模块
  -> Audit 模块

基础设施
  -> SQL Server：主业务库、配置库、审计库
  -> Redis：权限缓存、会话缓存、幂等标记
  -> MinIO/OSS：附件、报告、导入文件
  -> Hangfire：异步任务、智能体任务、数据扫描、报告生成
  -> 外部系统：MES、ERP、SCADA、质量系统、飞书、钉钉、LLM Provider
```

实现原则：

| 原则 | 说明 |
| --- | --- |
| 前端只做入口隔离 | 用户端和开发端页面分开，但安全边界由后端权限强制保证 |
| 后端统一鉴权授权 | 所有 API 经过 Authentication、Authorization、PermissionHandler |
| 数据配置化 | 数据源、表、字段、规则、工具、智能体都落到 SQL Server 配置表 |
| 任务异步化 | 智能体执行、数据扫描、报告生成、协同平台消息通过 Hangfire 执行 |
| 全链路审计 | 需求、权限、发布、模型、工具、输出全部记录 trace_id |

## 4. 用户端功能模块

用户端的设计目标是让业务人员不需要理解技术细节，也能完成需求提出、结果使用和效果反馈。

### 4.1 需求提报模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/enterprise/demands/new` |
| 后端接口 | `POST /api/enterprise/demands` |
| 主要数据 | 需求标题、业务场景、问题描述、期望指标、影响范围、附件 |
| 附件存储 | 文件上传到 MinIO/OSS，SQL Server 保存文件索引 |
| 状态变化 | `草稿 -> 已提交` |
| 输出 | 原始需求单、需求编号 |

用户可以选择质量、效率、成本、设备、排产等业务模板，也可以直接用自然语言描述问题。系统根据模板加载不同表单字段，字段必填、选项、校验规则都从配置表读取。

### 4.2 需求补充与确认模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/enterprise/demands/:id/confirm` |
| 后端接口 | `GET /api/enterprise/demands/{id}/summary`、`POST /api/enterprise/demands/{id}/confirm` |
| 主要数据 | 补充问题、补充附件、结构化需求摘要、确认意见 |
| 状态变化 | `需求整理中 -> 待用户确认 -> 已确认` |
| 输出 | 用户确认后的需求版本 |

开发端整理出的 `需求.md` 不直接作为用户阅读全文，而是在用户端展示摘要，包括业务目标、现状问题、涉及系统、预期指标、验收标准和风险前提。用户确认后，需求才能进入评估立项。

### 4.3 项目进度模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/enterprise/projects` |
| 后端接口 | `GET /api/enterprise/projects` |
| 主要数据 | 当前状态、处理人、预计时间、已完成节点、待用户处理事项 |
| 状态来源 | Workflow 状态机 |
| 输出 | 项目进度时间线 |

用户端只展示与用户相关的项目和状态，不展示开发端内部配置细节。

### 4.4 我的智能体模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/enterprise/agents` |
| 后端接口 | `GET /api/enterprise/agents` |
| 主要数据 | 已授权智能体、适用场景、最近使用时间、状态 |
| 权限来源 | 智能体授权表、角色权限、租户权限 |
| 输出 | 当前用户可使用的智能体列表 |

用户只能看到已经发布并授权给自己的智能体。未发布、测试中、无权限的智能体不会出现在列表中。

### 4.5 智能体问答模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | Web 问答页、飞书、钉钉 |
| 后端接口 | `POST /api/agent-runs`，协同平台走 Webhook |
| 执行方式 | Web 可同步创建任务，复杂任务进入 Hangfire 异步执行 |
| 权限控制 | 创建任务时校验智能体权限，执行时计算数据权限快照 |
| 输出 | 结论、依据、口径、权限说明、trace_id |

用户提问后，系统不把原始数据库结构全部交给大模型，而是先由 AgentRuntime 按用户权限和智能体绑定范围构建安全上下文。

### 4.6 结果查看与反馈模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/enterprise/agent-runs/:traceId`、反馈表单 |
| 后端接口 | `GET /api/agent-runs/{traceId}`、`POST /api/feedback` |
| 主要数据 | 回答内容、数据更新时间、调用规则、工具摘要、满意度、验收意见 |
| 输出 | 反馈记录、验收记录、效果评估基础数据 |

结果必须让用户知道三件事：结论是什么，依据是什么，哪些内容因为权限或数据不足没有展示。

## 5. 开发端功能模块

开发端的目标不是“写死一个 AI 功能”，而是把需求配置成可运行、可测试、可发布的智能体能力。

### 5.1 需求整理模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/demands/:id` |
| 后端接口 | `POST /api/developer/demands/{id}/documents` |
| 主要数据 | 原始需求、附件、补充问题、结构化需求文档 |
| 文件产出 | `需求.md` 保存为文档版本，正文可入库，附件入 MinIO/OSS |
| 输出 | 结构化需求、待用户确认摘要 |

`需求.md` 至少包含：

```text
背景和业务痛点
当前流程和问题现状
影响范围
涉及组织、产线、系统
期望指标
所需数据初步判断
业务规则初步判断
交付物
验收标准
风险与前提
```

### 5.2 数据评估模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/demands/:id/data-assessment` |
| 后端接口 | `POST /api/developer/data-assessments` |
| 主要数据 | 数据系统、数据对象、字段、时效、质量、权限风险 |
| 输出 | 数据评估报告、数据接入建议 |

数据评估要回答四个问题：

```text
需要哪些数据
这些数据在哪里
平台能不能拿到
拿到后哪些用户能看
```

### 5.3 数据源配置模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/data-sources` |
| 后端接口 | `POST /api/data-sources`、`POST /api/data-sources/{id}/test` |
| 支持方式 | API、直连数据库、同步数据库、文件导入、人工录入 |
| 安全要求 | 密码和 API Key 加密保存，企业数据库默认只读 |
| 输出 | 可用数据源配置 |

数据源配置成功后，可以触发 Hangfire 数据目录扫描任务，扫描库、表、字段元数据。

### 5.4 数据目录配置模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/data-catalog` |
| 后端接口 | `GET /api/data-catalog`、`PUT /api/data-catalog/fields/{id}` |
| 主要数据 | 表业务名称、字段业务名称、字段注释、单位、示例值、敏感级别 |
| 输出 | AI 可理解的数据目录 |

数据目录是连接企业数据和 AI 的关键。字段没有业务注释、没有敏感级别、没有权限配置时，不允许直接发布给智能体使用。

### 5.5 权限配置模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/admin/permissions`、`/developer/data-permissions` |
| 后端接口 | `POST /api/permissions`、`GET /api/permissions/effective-preview` |
| 权限范围 | 菜单、按钮、API、数据源、表、字段、工具、智能体 |
| 缓存策略 | Redis 缓存有效权限，权限变更后主动失效 |
| 输出 | 用户有效权限、权限预览、权限审计 |

权限计算规则：

```text
用户直接权限
  + 用户角色权限
  + 智能体绑定范围
  - deny 权限
  = 本次任务有效权限
```

智能体运行时取“用户权限”和“智能体绑定范围”的交集。交集之外的表、字段、工具不能进入 Prompt，也不能被工具执行。

### 5.6 业务规则配置模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/rules` |
| 后端接口 | `POST /api/rules`、`POST /api/rules/{id}/publish` |
| 规则类型 | 指标规则、口径规则、查询规则、映射规则、校验规则、输出规则、权限规则 |
| 版本控制 | 草稿、测试、发布、停用 |
| 输出 | 可被智能体调用的规则版本 |

规则分为系统规则和用户规则。系统规则用于平台通用约束，例如权限拦截、输出格式、敏感字段脱敏；用户规则用于企业或场景专属口径，例如良率计算、成本口径、设备状态解释。

### 5.7 工具配置模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/tools` |
| 后端接口 | `POST /api/tools`、`POST /api/tools/{id}/test` |
| 工具类型 | SQL 工具、HTTP 工具、Rule 工具 |
| 安全控制 | SQLGuard、只读账号、限行、超时、参数白名单、返回字段校验 |
| 输出 | 可授权、可审计、可测试的工具 |

工具执行前必须二次校验权限。即使智能体已经通过权限校验，工具运行时也要检查 SQL 涉及的表、字段和返回字段是否在本次权限快照内。

### 5.8 智能体配置模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/agents` |
| 后端接口 | `POST /api/agents`、`POST /api/agents/{id}/versions` |
| 配置内容 | 模型、提示词、数据范围、规则范围、工具范围、授权用户、输出格式 |
| 版本控制 | 草稿版本、测试版本、发布版本 |
| 输出 | 待测试或待发布智能体 |

智能体不是代码硬编码出来的，而是由配置组合出来：

```text
智能体 = 模型配置 + 提示词 + 授权数据目录 + 授权规则 + 授权工具 + 输出模板 + 使用权限
```

### 5.9 Workspace 沙箱测试模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/workspace` |
| 后端接口 | `POST /api/workspace/runs`、`POST /api/test-cases` |
| 测试类型 | 正常测试、异常测试、越权测试、边界测试、输出测试 |
| 关键能力 | 模拟不同用户权限、查看 Prompt 预览、查看工具调用链路 |
| 输出 | 测试记录、测试报告、发布建议 |

沙箱测试不能绕过权限。开发端在 Workspace 中模拟某个用户时，系统必须按该用户真实权限计算可见数据目录和可执行工具。

### 5.10 发布与交付报告模块

| 内容 | 实现方式 |
| --- | --- |
| 页面入口 | React `/developer/releases`、`/admin/agent-approvals` |
| 后端接口 | `POST /api/agents/{id}/submit-publish`、`POST /api/agents/{id}/approve` |
| 审核内容 | 数据范围、字段注释、权限范围、规则版本、工具测试、沙箱结果 |
| 输出 | 发布记录、上线说明、交付报告 |

交付报告应自动汇总需求、数据、规则、工具、智能体、测试记录、发布记录和验收反馈，形成可归档材料。

## 6. 用户端与开发端如何隔离

用户端和开发端既要隔离，又要通过流程中心协作。

### 6.1 前端隔离

| 隔离项 | 实现方式 |
| --- | --- |
| 路由隔离 | 用户端 `/enterprise/*`，开发端 `/developer/*`，管理端 `/admin/*` |
| 菜单隔离 | 菜单从后端权限接口加载，无权限不返回菜单项 |
| 页面组件隔离 | 用户端只显示需求、进度、智能体、结果、反馈；开发端显示配置、测试、发布 |
| 按钮隔离 | 新增、编辑、发布、审核等按钮根据按钮权限展示 |

前端隔离提升体验，但不作为安全依据。

### 6.2 后端隔离

| 隔离项 | 实现方式 |
| --- | --- |
| API 权限 | 每个接口绑定 permission_code，由 ASP.NET Core Authorization 校验 |
| 角色隔离 | 企业用户、开发者、FDE、管理员使用不同角色权限 |
| 服务编排隔离 | 用户端 API 只能调用用户用例，开发端 API 才能调用配置用例 |
| 数据返回隔离 | 后端按用户权限过滤字段和资源，不能依赖前端隐藏 |

### 6.3 数据隔离

| 隔离项 | 实现方式 |
| --- | --- |
| 租户隔离 | 所有业务表带 `tenant_id`，查询默认带租户条件 |
| 数据源隔离 | 数据源归属租户，跨租户不可见 |
| 表字段隔离 | 数据源、表、字段分层授权 |
| 智能体隔离 | 智能体归属租户和版本，授权用户才能使用 |
| 附件隔离 | MinIO/OSS object_key 与业务归属绑定，下载前鉴权 |

### 6.4 流程隔离

用户端能看到“需求进行到哪一步”，但看不到数据库连接串、SQL、Prompt 内部配置、模型密钥等开发细节。开发端能看到需求和配置任务，但不能代替用户确认验收结果。发布和验收通过流程中心连接两端。

```text
用户端提交需求
  -> 流程中心生成任务
  -> 开发端配置和测试
  -> 管理端审核发布
  -> 用户端使用和反馈
```

## 7. 核心模块交互

### 7.1 需求到立项交互

```text
用户端需求提报
  -> DemandService 保存需求单和附件索引
  -> WorkflowService 创建流程实例
  -> 开发端 DemandDocumentService 生成需求.md
  -> 用户端确认需求摘要
  -> WorkflowService 流转到评估中
  -> ProjectService 创建项目或实施任务
```

关键落库对象：

```text
demand_records
demand_documents
demand_attachments
workflow_instances
workflow_tasks
project_records
audit_operation_logs
```

### 7.2 数据接入与目录交互

```text
开发端配置数据源
  -> DataSourceService 加密保存连接信息
  -> Hangfire 创建 DataCatalogScanJob
  -> Adapter 读取数据库/API/文件元数据
  -> DataCatalogService 写入表和字段目录
  -> 开发端补充字段注释、单位、敏感级别
  -> 权限中心配置字段授权
```

关键控制点：

| 控制点 | 说明 |
| --- | --- |
| 连接测试 | 保存前测试数据源可用性 |
| 只读原则 | 企业数据库账号默认只读 |
| 字段注释 | 注释不足时不允许发布给智能体 |
| 敏感级别 | sensitive/confidential 字段默认不进入 AI 上下文 |
| 审计 | 数据源新增、修改、扫描都记录审计 |

### 7.3 权限与智能体交互

```text
用户发起智能体任务
  -> AgentRunService 校验用户是否可使用智能体
  -> PermissionService 计算用户有效权限
  -> AgentLoader 加载智能体发布版本
  -> 取用户权限和智能体绑定范围交集
  -> 保存 PermissionSnapshot
  -> ContextBuilder 只注入授权数据目录、规则、工具
```

交互重点：

```text
无权限字段不进入 Prompt
无权限工具不进入可调用工具列表
无权限数据不出现在模型输出中
工具执行前再次校验权限快照
AnswerGuard 对最终输出做权限和敏感信息检查
```

### 7.4 规则、工具与智能体交互

```text
开发端发布规则
  -> RuleEngine 保存发布版本
  -> 智能体绑定规则版本
  -> Workspace 测试规则执行结果

开发端发布工具
  -> ToolRuntime 保存工具 Schema
  -> 工具绑定数据源和权限范围
  -> SQLGuard / HttpGuard / RuleGuard 执行测试

智能体运行
  -> AgentRuntime 根据用户问题选择规则和工具
  -> ToolRuntime 执行授权工具
  -> RuleEngine 计算业务口径
  -> AgentRuntime 汇总结果并生成回答
```

这里的实现重点是：规则和工具必须有版本，智能体绑定的是明确版本，不能因为后续规则修改导致历史运行结果不可解释。

### 7.5 Workspace 到发布交互

```text
开发端提交测试
  -> WorkspaceRunService 创建测试任务
  -> AgentRuntime 使用模拟用户权限执行
  -> AuditService 记录 Prompt、工具、输出摘要
  -> TestReportService 汇总测试结果
  -> 发布检查通过后提交审核
  -> 管理端审核
  -> AgentPublishService 固化发布版本
```

发布检查项：

| 检查项 | 要求 |
| --- | --- |
| 数据范围 | 至少绑定必要数据源、表、字段 |
| 字段注释 | 关键字段有业务注释、单位、口径 |
| 权限范围 | 配置使用用户或角色，不能默认全员 |
| 规则版本 | 绑定已发布规则 |
| 工具测试 | 工具测试通过，失败场景有提示 |
| 越权测试 | 无权限字段不进入 Prompt 和输出 |
| 输出检查 | 有结论、依据、口径、权限说明、trace_id |

### 7.6 用户使用与审计交互

```text
Web/飞书/钉钉提问
  -> 创建 AgentTask
  -> Hangfire Worker 执行
  -> AgentRuntime 调模型和工具
  -> AnswerGuard 检查输出
  -> AuditService 写入 trace 日志
  -> 用户端返回结果
  -> 用户提交反馈
  -> DeliveryReportService 汇总交付效果
```

每次任务必须能通过 `trace_id` 查到：

```text
谁发起的任务
使用哪个智能体和版本
当时权限快照是什么
注入了哪些数据目录
调用了哪个模型
执行了哪些工具
用了哪些规则
输出了什么摘要
是否发生权限拦截、重试、降级
```

## 9. 可实现性验收标准

| 验收项 | 标准 |
| --- | --- |
| 端隔离 | 用户端、开发端、管理端路由和菜单隔离，后端 API 强校验 |
| 需求闭环 | 用户可提交需求，开发端可生成需求文档，用户可确认 |
| 数据配置 | 数据源、表、字段、注释、敏感级别可配置 |
| 权限控制 | 数据源、表、字段、工具、智能体权限可配置且运行时生效 |
| 规则配置 | 系统规则和用户规则可配置、可发布、可绑定智能体 |
| 工具安全 | SQL/HTTP/Rule 工具可测试、可审计，SQL 工具只读、限行、超时 |
| 智能体配置 | 智能体可配置模型、提示词、数据、规则、工具和授权范围 |
| 沙箱测试 | 可模拟不同用户验证权限、Prompt、工具调用和输出 |
| 发布审核 | 未测试或审核不通过的智能体不能上线 |
| 运行审计 | 每次智能体执行都有 trace_id，可追踪权限、模型、工具、输出 |
| 高可用 | API 无状态、Worker 异步任务、Redis 缓存、SQL Server 备份、外部依赖降级 |

## 10. 结论

重新整理后的方案重点不是继续增加概念，而是把系统拆成可实现的端、模块、接口、数据和流程。

本系统第一版应优先完成：

```text
用户端能提需求、确认需求、使用智能体、反馈效果
开发端能整理需求、配置数据、配置规则、配置工具、配置智能体、完成测试发布
权限中心能保证字段级权限和工具权限在运行时生效
智能体运行中心能保证每次执行有依据、有边界、有审计
流程中心能把需求、开发、测试、发布、使用、反馈串成闭环
```

一句话总结：

```text
这是一个把电子信息制造业企业的模糊业务需求，转化为受控、可测、可发布、可审计智能体能力的平台。
```
